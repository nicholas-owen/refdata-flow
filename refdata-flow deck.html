Motivation

The reference data problem.

Every analysis starts the same way — somebody downloads a genome. And then somebody else downloads it again.

  • 01The same FASTA fetched five times across a single shared filesystem.
  • 02Different groups quietly running against different Ensembl builds.
  • 03“Where did this come from?” answered by reading someone’s bash_history.
  • 04No way to prove, in a paper, exactly which bytes were used.

Architecture

Three steps, end to end.

01 · Resolve
Map generic queries to exact assemblies.
resolve.py · Python
Searches Ensembl, UCSC and NCBI; prompts on ambiguity; caches answers to a resolved CSV.
02 · Download
Parallel fetches, on background nodes.
main.nf · Nextflow · genomepy
One job per assembly. FASTA and GTF land compressed in provider-specific staging directories.
03 · Ingest
Hash, alias, archive.
update_refgenie.py · refgenie v3
Refgenie computes Refget hashes, builds the content-addressed vault, wires up aliases, and cleans up.
requests_resolved.csv
provider/species/assembly/*.gz

01

Step 1

Resolution.

From “hg38” to an exact assembly, before a single byte is downloaded.

  • ↳Queries the three big providers. Ensembl, UCSC, NCBI. Whichever matches first wins — or all of them, if you leave provider blank.
  • ↳Interactive on ambiguity. Multiple matches for “mouse”? You pick once. The choice is written to disk.
  • ↳Cached forever. A second run skips the network for anything already pinned in requests_resolved.csv.
  • ↳Reversible. Want a different assembly? Delete that one row and the wizard re-prompts.

requests_raw.csv

# species, query, provider, annotation, aliases
Homo_sapiens,grch38,ensembl,true,hg38;human
Mus_musculus,mouse,,true,mm39;mm
Saccharomyces_cerevisiae,sacCer3,ucsc,false,yeast

Empty provider ⇒ search them all. Semicolons in aliases.

02

Step 2

Download.

One Nextflow process, one genomepy job per row, fanned out across the cluster.

  • ↳Native parallelism. Each resolved row is its own conda-isolated job; Nextflow handles scheduling.
  • ↳Always compressed. bgzip-encoded FASTA and GTF land directly — no intermediate decompression.
  • ↳Resume-safe. Re-run after adding one species; only that species moves.
  • ↳Provenance per dataset. Each output ships with README.txt and a custom provenance.json.

main.nf · the only process

process FETCH_GENOME {
    tag        "${species}_${assembly}"
    publishDir "${outdir}/${provider}/${species}"

    input:
      tuple val(species), val(assembly),
            val(provider), val(annotation)

    script:
    """
    python3 bin/download.py \
      --species "${species}" --assembly "${assembly}" \
      --provider "${provider}" ${annotation_flag}
    """
}

03

Step 3

Refgenie ingestion.

Where loose downloads become a content-addressed, alias-rich, permanent vault.

  • ↳Self-initialising. First run? refgenie init happens automatically.
  • ↳Cryptographic hashes. refgenie build reads every base, computes a Refget hash, stores by hash.
  • ↳Aliases as symlinks. Your CSV aliases — hg38, human — become first-class lookup paths.
  • ↳Zero-duplication cleanup. Once the vault has the bytes, the Nextflow staging copy is removed.

what gets written

# content-addressed vault
data/2230c53b…/fasta/default/GRCh38.p14.fa.gz
data/2230c53b…/annotation/default/…gtf.gz

# human-readable shortcuts
alias/GRCh38.p14/  →  data/2230c53b…/
alias/grch38/      →  alias/GRCh38.p14/
alias/hg38/        →  alias/GRCh38.p14/
alias/human/       →  alias/GRCh38.p14/

From the user side

One command, end to end.

Edit a CSV, run the wrapper, point it at storage. Everything else is automatic.

typical run

# 1. edit requests_raw.csv with the species you want

# 2. launch end-to-end on the HPC
bash run_pipeline.sh requests_raw.csv \
     --outdir /reference_data/

# resolver → nextflow → refgenie, all in one go

useful flags

  • --dry-runResolve and report. Touch no genomes.
  • --cleanSweep work/, .nextflow/, logs. Vault untouched.
  • --outdir <path>Override the destination. Defaults to data/references.
  • --versionPrint the build & exit.

    ├── grch38/      → alias/GRCh38.p14/

  • ↳The vault is the truth. Files keyed by Refget hash. Bytes never move once written.
  • ↳Aliases are the doorway. Official names, CSV queries, and custom aliases all resolve to the same vault entry.
  • ↳Multiple ways in, one source. Three users asking for hg38, grch38, GRCh38.p14 — all point at the same physical file.
  • ↳Per-dataset provenance. A README.txt and provenance.json sit alongside each entry.

On disk

Storage stays lean by design.

01 — Always compressed
bgzip end‑to‑end.
Genomes never sit uncompressed on disk. Genomepy writes .fa.gz / .gtf.gz natively.
~3 GB → ~900 MB per human genome
02 — Resume, don’t redo
Nextflow remembers everything.
Add one species to the CSV next month; only that one runs. The other nine are skipped instantly.
-resume is on by default
03 — One copy, not two
Staging is cleared after ingest.
Once refgenie has the bytes in the hash vault, the Nextflow staging directory is removed.
--clean wipes work/ & logs on demand

The headline feature

A fingerprint for every genome.

A Refget hash — 2230c53b… — is a mathematical signature of the sequence itself. Not a filename. Not a URL. Not a version string. The bytes.

What it gives you
Two researchers, two HPCs, identical FASTA — identical hash. Mismatched hash means mismatched data, end of discussion.
Why it matters
A paper can cite the hash. A reviewer five years later can verify the exact sequence used — the closest genomics has to a DOI for data.
How it’s computed
Refgenie v3 reads every base on ingest, applies the GA4GH Refget standard, and stores the result in the vault index.
